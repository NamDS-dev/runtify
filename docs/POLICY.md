## 🟡 1. 이메일 인증(Verification) 정책

> 관련 구현: [FEATURE_PLAN.md — 이메일 인증 플로우](FEATURE_PLAN.md)

### 배경
- 현재 이메일 회원가입 시 인증 메일을 발송하지 않음 → 미검증 계정도 모든 기능 사용 가능
- OAuth(Google/Apple)는 인증된 이메일이 제공되므로 별도 검증 불필요

### 결정 필요 사항 (2026-04-23 확정)

| 항목 | 결정 |
|------|------|
| **미인증 상태에서 허용할 기능** | ✅ **러닝 시작 1회까지 허용** (모든 기능 view 가능) |
| **미인증 상태에서 차단할 기능** | ✅ **2회차 러닝 종료 시점 + 모든 기능 작동(크루 가입·리워드·랭킹 등록)** |
| **인증 메일 재발송 쿨다운** | ✅ **슬라이딩 윈도우: 5분 / 최대 3회** (3회 도달 시 첫 발송 타임스탬프 + 5분까지 차단, 시간 지나면 슬롯 회수) |
| **인증 유예 기간** | ✅ **러닝 1회까지** (위 항목으로 자동 정의됨) |
| **OAuth 가입자의 `emailVerified`** | ✅ **항상 true** (Google/Apple 검증 간주) |

### 구현 전제 (확정 사항 반영)
- `UserModel`에 `emailVerified: bool` 필드 추가
- 미인증 사용자 흐름:
  - 가입 → 인증 메일 자동 발송
  - 모든 화면 view 가능 (홈/크루/랭킹/리워드 둘러보기 OK)
  - 러닝 시작 가능 → 1회는 정상 진행
  - **러닝 종료 시점에 인증 체크** → 미인증이면 저장 차단 + "인증하시겠어요?" 다이얼로그
  - 크루 가입/리워드 교환/랭킹 등록 등 **기능 작동 시 동일한 인증 유도 다이얼로그**
- 인증 유도 다이얼로그 패턴:
  ```
  ┌──────────────────────────────┐
  │  📧 이메일 인증이 필요해요    │
  │                              │
  │  러닝 기록을 저장하려면        │
  │  이메일 인증을 완료해주세요    │
  │                              │
  │  [ 인증 메일 재발송 ]         │
  │  [ 나중에 ]                   │
  └──────────────────────────────┘
  ```
- 러닝 데이터는 인증 완료 후 저장 가능하도록 **로컬 임시 보관** (사용자가 인증 완료 시점에 자동 동기화)

## 🟡 2. 로그인 실패 레이트 리밋 정책

> 관련 구현: [FEATURE_PLAN.md — 레이트 리밋](FEATURE_PLAN.md)

### 배경
- 현재 Firebase 기본 rate limit(IP 기반)에만 의존 → 계정 단위 브루트포스 방어 없음
- 로컬만 구현 시 앱 재설치로 우회 가능

### 결정 필요 사항 (2026-04-23 확정)

| 항목 | 결정 |
|------|------|
| **실패 허용 횟수** | ⬜ 구현 시 디폴트 **3회** (변경 의견 있을 시 결정) |
| **잠금 시간** | ⬜ 구현 시 디폴트 **60초 고정** (점진 증가는 차후 검토) |
| **서버 측 강화 여부** | ✅ **MVP는 로컬만**, 서버(Cloud Functions)는 출시 전 별도 검토 |
| **Blaze 요금제 전환** | ✅ **출시 직전 결정** — FEATURE_PLAN 출시 로드맵에 등록 |
| **성공 시 카운터 리셋** | ⬜ 구현 시 디폴트 **즉시 리셋** (단순함 우선) |

### 구현 전제 (Phase 1 — 로컬만)
- 이메일별 키로 `SharedPreferences` 저장 (예: `login_fail_count_{email_hash}`)
- UI에 남은 대기 시간 카운트다운 표시
- 한계 인식: 앱 재설치 시 우회 가능 → Phase 2(서버) 출시 전 보강 예정

## 🟡 3. 세션 만료 및 토큰 갱신 정책

> 관련 구현: [FEATURE_PLAN.md — 세션 만료 처리](FEATURE_PLAN.md)

### 배경
- Firebase ID token은 1시간마다 자동 갱신 시도 → 명시적 갱신 실패 대응 없음
- 러닝 중 토큰 만료 시 UX 시나리오가 가장 까다로움 (데이터 유실 가능)

### 결정 필요 사항 (2026-04-23 확정)

| 항목 | 결정 |
|------|------|
| **유휴 강제 재인증** | ✅ **적용 안 함** (러닝 중심 앱이라 사용 패턴 불규칙) |
| **러닝 중 토큰 만료 UX** | ✅ **러닝 중 절대 로그아웃 X** — 앱 강제 종료 외에는 세션 유지 |
| **러닝 데이터 저장 시점** | ✅ **러닝 종료 버튼 시점에만 저장** (중간 저장 X) |
| **토큰 갱신 실패 처리** | ✅ Firebase SDK 기본 자동 갱신에 의존, 명시적 추가 처리 없음 |
| **러닝 종료 시 토큰 만료된 경우** | ✅ 로컬 큐잉 + 다음 온라인 복귀 시 자동 동기화 (러닝 데이터는 절대 유실되지 않음) |

### 구현 전제 (확정 사항 반영)
- **별도 SessionManager 불필요** — Firebase SDK의 자동 토큰 갱신(1시간 간격)에 위임
- `RunningProvider`에 `isRunningInProgress` 플래그 추가 → `idTokenChanges()` 리스너에서 이 플래그 true일 땐 로그아웃 트리거 무시
- 러닝 저장 실패 시 (네트워크/토큰 등 모든 사유) 로컬 SharedPreferences/Hive에 queue로 보관
  - 앱 재실행 또는 온라인 복귀 시 큐 자동 flush
  - 사용자에겐 "동기화 중..." 표시
- **전제 검증 필요**: 현재 `running_firestore_datasource.dart`의 `saveSession`이 이미 종료 버튼 시점 1회 호출인지 확인 (코드 점검 필요)

## 🔴 4. 계정 탈퇴 정책

> 관련 구현: [FEATURE_PLAN.md — 계정 탈퇴 플로우](FEATURE_PLAN.md)
> App Store 14+ 정책상 **필수** — 미구현 시 심사 거절 가능성

### 배경
- 현재 프로필 페이지에 로그아웃만 있음. 탈퇴 기능 전무
- 크루 리더 탈퇴, 랭킹 기여 데이터 등 연관 데이터가 많아 정책 결정이 가장 복잡

### 결정 필요 사항 (2026-04-26 부분 확정)

| 항목 | 결정 |
|------|------|
| **연관 데이터 처리** | ✅ **소프트 삭제** (탈퇴 즉시 숨김 → 30일 후 하드 삭제) |
| **러닝 기록 처리** | ✅ 소프트 삭제와 동일 — 30일간 숨김 후 일괄 삭제 |
| **크루 리더 탈퇴 시** | ✅ **탈퇴 제한** — 다른 멤버에게 수동 위임 후에만 탈퇴 가능 (가이드 UI 필수) |
| **유예 기간** | ✅ **30일 복구 가능** (소프트 삭제 기간) |
| **재인증 요구** | ✅ **이중 계층** — 1차 비밀번호(소셜 가입자=소셜 재로그인) + 2차 이메일 6자리 인증 코드. 둘 다 통과 시에만 탈퇴 진행 |
| **GDPR/CCPA 문구** | ✅ **국내 출시 후 해외 시장 진입 시점에 추가** (MVP에선 제외) |
| **배지·업적 처리** | ⬜ 구현 시 디폴트 **소프트 삭제와 동일** (30일 숨김 후 삭제). 크루 이력 보존 필요 시 재논의 |

### 구현 전제 (확정 사항 반영)
- `users/{uid}` 문서에 `deletedAt: timestamp?`, `scheduledHardDeleteAt: timestamp?` 필드 추가
- 모든 조회 쿼리에서 `where('deletedAt', '==', null)` 필터 추가 (또는 클라이언트 필터)
- 크루 리더 탈퇴 시도 시: 멤버 1명 이상이면 "다른 멤버에게 리더 양도 후 탈퇴 가능" 안내, 본인 외 멤버 0명이면 즉시 탈퇴 + 크루 자동 해체
- 30일 후 하드 삭제는 **Cloud Functions 스케줄러** 필요 → Blaze 요금제 시점(레이트 리밋 Phase 2와 묶어서 출시 2주 전)
- 30일 내 재로그인 시 복구 화면 — "탈퇴 처리 중인 계정입니다. 복구하시겠습니까?"
- 프로필 페이지에 "계정 삭제" 버튼 + 확인 다이얼로그 + 비밀번호 재입력
- `FirebaseAuth.currentUser.delete()`는 30일 후 Cloud Functions에서 호출 (즉시 호출 X)
